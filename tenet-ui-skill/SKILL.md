# Tenet UI skill

Turn a design into **tenet-ui-compliant React**, verify it renders, and audit
existing code — grounded in the *current* component API and tokens (ingested
under `knowledge/`), not memory.

> Configured design system: **Tenet UI** ("Editorial Ink"), package `tenet-ui`,
> in `demo-design-system/`. The knowledge base is refreshed by
> `node scripts/ingest.mjs` (copies the package's `dist/tokens.json`,
> `generated/components.json`, and `guidelines/`). Re-run it after rebuilding the
> package so the skill never drifts from the real API.

## When to use
- "Build this screenshot / Figma frame in React" → **Generate**.
- "Is this component compliant with our design system?" / "find violations" → **Audit**.
- "Does my implementation match the design?" → **Verify**.

## Knowledge base (read on demand — do not inline whole files)
Under `knowledge/` (generated by `scripts/ingest.mjs`):
- `knowledge/runtime.json` — **the runtime contract: which stylesheet to import so
  tokens AND component styles resolve. Read this FIRST when generating — nothing
  renders until it's wired.**
- `knowledge/tokens.json` — every legal token: name, CSS var, resolved light/dark
  value (or shared value), description.
- `knowledge/components.json` — every component with its real props, defaults,
  sub-components, import path (`tenet-ui`).
- `knowledge/usage/<name>.md` — per-component usage guidelines (when to use, rules,
  do/don't). Load the few relevant to what you're building.

Load only the entries relevant to the current task.

## A design system is a RUNTIME contract, not just a token dictionary
This is the lesson the previous skill was rebuilt around — do not lose it.

Citing real tokens is necessary but **not sufficient**. tenet-ui ships its tokens
**and** its component styles as one stylesheet (`tenet-ui/styles.css`). Every
`var(--token)` and every `Tenet-*` class is undefined until that file is imported
once at the app root. Theming is just `data-color-mode="light|dark"` on an
ancestor — there is no provider to mount. Code can reference every token and
component correctly and still render **completely unstyled** — no background, no
borders, fallback fonts, collapsed spacing, broken controls — because nothing
loaded the styles. **A clean content audit does NOT mean it renders. Verify is the
gate.**

## Workflows
1. **Generate** → `reference/generate.md`
2. **Verify** → `reference/verify.md`
3. **Audit** → `reference/audit.md`

## The compliance rubric (judge the CODE, not the look)
Two outputs can render pixel-identical while one is non-compliant — *and* a
token-perfect file can render broken. Always check, in this order:

- **Runtime wiring** — `tenet-ui/styles.css` is imported so tokens + component
  styles resolve. Check with `node scripts/audit-runtime.mjs <project-root>`.
  **If this fails, nothing below matters — the screen is unstyled.**
- **Components** — the real tenet-ui component for the role (a status pill →
  `Label`, a toggle → `Switch`, a dropdown → `Select`, an action overflow →
  `Menu`, …), never a raw `<button>`/`<input>` or a reinvented styled `<div>`.
- **Color** — a semantic token (`--fgColor-accent`, `--bgColor-default`, a status
  tint), never a hardcoded hex/rgb/named color.
- **Spacing & size** — `--space-*` / `--base-size-*` / `--control-*-size`, never
  arbitrary px.
- **Typography / radius / border / shadow** — tokens
  (`--fontFamily-display`/`-body`, `--borderRadius-*`, `--borderWidth-*`,
  `--shadow-*`).
- **API** — real props + valid values from `knowledge/components.json` (e.g.
  `variant="primary"`, not an invented prop).
- **Guidelines** — follows the rules in `knowledge/usage/<name>.md` (esp. the
  accessible-name requirements).

Two scanners enforce this: `scripts/audit-runtime.mjs` (does the DS resolve at
runtime?) and `scripts/audit.mjs <file>` (is each file token/component/API-clean?).
Run **both**. Any fix re-scores to zero on both — then **Verify** confirms it
actually renders.

## Worked example — why "looks right" lies (the canonical demo)
Two agents rebuilt the same GitHub-style page:
- **No skill:** looked pixel-perfect — because it *hardcoded* the whole palette
  (`#0d1117`, `#30363d`, …) into a hand-rolled `Box` and reinvented the controls.
  Passes the runtime check, **fails the content audit hard.** A theme switch never
  reaches it. This is the anti-pattern the skill exists to prevent.
- **With skill (before the fix):** looked broken (unstyled) — because it
  referenced real tokens/components but **never imported the stylesheet**, so every
  `var(--token)` and `Tenet-*` class was undefined. **Passed `audit.mjs`, failed
  `audit-runtime.mjs`.** One import from being both correct and compliant.

Lesson: optimize for token *resolution* (runtime) **and** token *hygiene*
(content), and gate on **Verify** — not on the look, and not on a single audit.
