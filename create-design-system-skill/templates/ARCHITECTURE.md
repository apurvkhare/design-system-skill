## What this skill does
Turns a design (screenshot / Figma / mockup) into React that complies with
{{DESIGN_SYSTEM_NAME}}, audits existing code for violations, and verifies that the
result actually renders. It is grounded in an **ingested snapshot** of the real API
under `knowledge/`, not in the model's training-data memory — so it stays correct as
the design system evolves.

## The core insight it encodes
A design system is a **runtime contract**, not just a token dictionary. Referencing
real tokens/components is necessary but not sufficient: nothing renders until the
stylesheet is imported / provider mounted (`knowledge/runtime.json`). The skill
therefore audits on two independent axes — *content* (are the right tokens/components
used?) and *runtime* (do they resolve?) — and gates on a real render (Verify).

## How it works (runtime)
- **`design-system.config.json`** — the single source of truth for where the API
  comes from and how the DS is wired. Retarget by editing this and re-running ingest.
- **`scripts/ingest.mjs`** — config-driven; writes `knowledge/{tokens,components,
  usage,icons,runtime,meta}.json`. Re-run whenever the design system changes.
- **`scripts/audit.mjs <file>`** — per-file content audit: hardcoded colors, off-scale
  px, unknown tokens, raw elements with a DS equivalent, inline icons, invented props.
- **`scripts/audit-runtime.mjs <project>`** — the runtime gate: is the stylesheet
  imported / provider mounted so the DS resolves? Catches the #1 failure mode.
- **`reference/{generate,audit,verify}.md`** — the three workflows.
- **`knowledge/`** — the ingested snapshot the model reads on demand.
