# {{DESIGN_SYSTEM_NAME}} skill

Turn a design into **{{DESIGN_SYSTEM_NAME}}-compliant React**, verify it renders, and
audit existing code — grounded in the *current* component API and tokens (ingested
under `knowledge/`), not memory.

> Configured design system: **{{DESIGN_SYSTEM_NAME}}**, package `{{PACKAGE}}`.
> The knowledge base is refreshed by `node scripts/ingest.mjs` (driven by
> `design-system.config.json`). Re-run it when the design system changes so the
> skill never drifts from the real API. See `ARCHITECTURE.md` for how this skill
> was built and how each piece works.

## When to use
- "Build this screenshot / Figma frame in React" → **Generate**.
- "Is this component compliant with our design system?" / "find violations" → **Audit**.
- "Does my implementation match the design?" → **Verify**.

## Knowledge base (read on demand — do not inline whole files)
Under `knowledge/` (generated by `scripts/ingest.mjs`):
- `knowledge/runtime.json` — **the runtime contract: which stylesheet(s) to import
  (and provider(s) to mount) so tokens AND component styles resolve. Read this FIRST
  when generating — nothing renders until it's wired.**
- `knowledge/tokens.json` — every legal token: name, CSS var, resolved light/dark
  value (or shared value), description.
- `knowledge/components.json` — every component with its real props, defaults,
  sub-components, import path (`{{PACKAGE}}`).
- `knowledge/usage/…` — per-component usage guidelines (when to use, rules, do/don't).
  Load only the few relevant to what you're building.
{{ICONS_KNOWLEDGE_LINE}}

## A design system is a RUNTIME contract, not just a token dictionary
Citing real tokens is necessary but **not sufficient**. Every `var(--token)`
(and every component style) is undefined until the design system's stylesheet is
imported / its provider mounted at the app root — exactly as `knowledge/runtime.json`
describes. Code can reference every token and component correctly and still render
**completely unstyled** — no background, no borders, fallback fonts, collapsed
spacing, broken controls — because nothing loaded the styles. **A clean content
audit does NOT mean it renders. Verify is the gate.**

## Workflows
1. **Generate** → `reference/generate.md`

## The compliance rubric (judge the CODE, not the look)
Two outputs can render pixel-identical while one is non-compliant — *and* a
token-perfect file can render broken. Always check, in this order:

- **Runtime wiring** — the required stylesheet(s)/provider(s) from
  `knowledge/runtime.json` are present so tokens + component styles resolve.
  Check with `node scripts/audit-runtime.mjs <project-root>`.
  **If this fails, nothing below matters — the screen is unstyled.**
- **Components** — the real {{DESIGN_SYSTEM_NAME}} component for the role (a status
  pill → the tag/label component, a toggle → the switch, a dropdown → the select, an
  action overflow → the menu, …), never a raw `<button>`/`<input>` or a reinvented
  styled `<div>`.
- **Color** — a semantic color token, never a hardcoded hex/rgb/named color.
- **Spacing & size** — spacing/size tokens, never arbitrary px.
- **Typography / radius / border / shadow** — tokens, not literals.
{{ICONS_RUBRIC_LINE}}
- **API** — real props + valid values from `knowledge/components.json` (never an
  invented prop).
- **Guidelines** — follows the rules in `knowledge/usage/…` (esp. accessible-name
  requirements).

Two scanners enforce this: `scripts/audit-runtime.mjs` (does the DS resolve at
runtime?) and `scripts/audit.mjs <file>` (is each file token/component/API-clean?).
Run **both**. Any fix re-scores to zero on both — then **Verify** confirms it
actually renders.

## Refreshing the knowledge base
When {{DESIGN_SYSTEM_NAME}} ships changes, re-run `node scripts/ingest.mjs` so the
skill tracks the current API. The ingest sources are declared in
`design-system.config.json`; `knowledge/meta.json` records provenance and counts.
